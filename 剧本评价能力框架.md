# 剧本评价能力 - 完整代码框架

本文档包含三种审稿模式的完整框架：
1. **全本通审 (Deep Audit)** - 对整本剧本进行宏观评估，输出7维度评分和批注
2. **分集精修 (Episode Audit)** - 对单集剧本进行精细审核，输出问题列表和修改指令
3. **横向对比 (Script Comparison / AB Test)** - 对多版本剧本进行横向测评，选出最佳版本

```typescript
// 单条审稿批注（全本模式）
interface AuditAnnotation {
  id: string;
  dimension: string;           // 维度，如 "节奏.拖沓", "逻辑.动机"
  location: {
    lineContent: string;       // 剧本中的原文（用于高亮定位）
    scene?: string;            // 场次，如 "第3集 3-2场"
  };
  score: number;               // 0=致命伤(Fatal), 1=建议优化(Warning)
  issue: string;               // 问题描述
  suggestion: string;          // 修改建议
  canBatchFix: boolean;        // 是否可批量修复
  status: 'pending' | 'fixed' | 'ignored';
}

// 维度评分
interface AuditScore {
  dimension: string;           // 维度名称
  score: number;               // 0-100分
  comment: string;             // 评分说明
}

// 完整审稿报告（全本模式）
interface AuditReport {
  scriptId: string;
  totalScore: number;          // 总分 0-100
  dimensionScores: AuditScore[];
  annotations: AuditAnnotation[];
  summary: string;             // 总体评价
  timestamp: number;
}

// -------- 分集精修 (Episode Audit) 类型 --------

// 单条审稿问题（分集模式）
interface AuditItem {
  id: string;
  category: string;            // 问题类别：主线/衔接/节奏/冲突/人设/视觉/台词
  severity: 'high' | 'medium' | 'low';  // 严重程度
  issue: string;               // 问题描述
  targetQuote: string;         // 剧本中存在问题的原文（用于定位）
  suggestion: string;          // 具体修改指令
  status: 'pending' | 'applied' | 'ignored';
}

// 分集审稿配置
interface EpisodeAuditConfig {
  novelContent: string;        // 本集对应的小说原文
  outlineContent: string;      // 本集大纲
  previousScriptContext: string; // 上一集结尾（用于衔接检查）
  episodeId: number;           // 集数
  episodeTitle: string;        // 集标题
  worldSetting?: string;       // 世界观设定
  keyQuotes?: string;          // 关键金句
  characterList?: CharacterProfile[]; // 人物设定列表
  openingHook?: string;        // 开场钩子要求
  endingHook?: string;         // 结尾钩子要求
  foreshadowing?: string;      // 伏笔要求
  targetChapter?: string;      // 对应原著章节
}

interface CharacterProfile {
  id: string;
  name: string;
  desc: string;                // 人物描述（含状态、心理、关系）
}
```

```typescript
// ============ API 配置 ============

interface APIConfig {
  apiKey: string;
  baseUrl: string;
  model: string;
}

// 通用 API 调用函数
async function callLLMAPI(
  config: APIConfig,
  messages: { role: string; content: string }[],
  options: {
    temperature?: number;
    responseFormat?: { type: 'json_object' | 'text' };
    timeout?: number;
  } = {}
): Promise<{ text: string }> {

  const { temperature = 0.4, responseFormat, timeout = 180000 } = options;

  // OpenAI 兼容格式
  const response = await fetch(`${config.baseUrl}/chat/completions`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${config.apiKey}`
    },
    body: JSON.stringify({
      model: config.model,
      messages: messages,
      temperature: temperature,
      response_format: responseFormat
    }),
    signal: AbortSignal.timeout(timeout)
  });

  const data = await response.json();
  return { text: data.choices[0].message.content };
}
```

## 三、核心评价函数

### 3.1 全本通审函数 (Deep Audit)

```typescript
// ============ 剧本深度评价函数 ============

export async function evaluateScript(
  apiConfig: APIConfig,
  script: string,                    // 待评价的剧本
  novelContext: string = "",         // 原著参考（可选）
  outlineContext: string = ""        // 大纲参考（可选）
): Promise<AuditReport> {

  // 1. 构建评价 Prompt
  const prompt = buildEvaluationPrompt(script, novelContext, outlineContext);

  // 2. 根据剧本长度设置超时
  const timeout = calculateTimeout(script.length);

  // 3. 调用 API
  const response = await callLLMAPI(
    apiConfig,
    [{ role: 'user', content: prompt }],
    {
      temperature: 0.4,
      responseFormat: { type: 'json_object' },
      timeout: timeout
    }
  );

  // 4. 解析返回结果
  const result = JSON.parse(cleanJsonResponse(response.text));

  // 5. 格式化为标准报告
  return formatAuditReport(result);
}

// 计算超时时间
function calculateTimeout(scriptLength: number): number {
  if (scriptLength > 100000) return 900000;      // 15分钟
  if (scriptLength > 50000) return 480000;       // 8分钟
  if (scriptLength > 20000) return 300000;       // 5分钟
  return 180000;                                  // 3分钟
}

// 清理 JSON 响应
function cleanJsonResponse(text: string): string {
  // 移除 markdown 代码块标记
  let cleaned = text.replace(/```json\s*/gi, '').replace(/```\s*/gi, '');
  // 移除首尾空白
  cleaned = cleaned.trim();
  return cleaned;
}

// 格式化审稿报告
function formatAuditReport(raw: any): AuditReport {
  return {
    scriptId: Date.now().toString(),
    totalScore: raw.totalScore || 0,
    dimensionScores: raw.dimensionScores || [],
    annotations: (raw.annotations || []).map((a: any, i: number) => ({
      ...a,
      id: `${Date.now()}-${i}`,
      status: 'pending'
    })),
    summary: raw.summary || '',
    timestamp: Date.now()
  };
}
```

## 四、完整评价 Prompt（核心指令词）

### 4.1 全本通审 Prompt (Deep Audit Prompt)

```typescript
function buildEvaluationPrompt(
  script: string,
  novelContext: string,
  outlineContext: string
): string {

  return `
# Role
你是一名**资深短剧改编顾问**，专注于**小说IP改编短剧**赛道。
请对以下【剧本正文】进行深度体检，核心任务是评估**小说到短剧的改编质量**，包括剧情压缩、节奏把控、情绪放大等改编技巧。

# Priority Rules (评估优先级)
1. **改编视角优先**：始终以"小说原文"为参照基准，评估剧本的改编取舍是否合理。
2. **短剧特性优先**：短剧需要快节奏、强情绪、高密度，评估时需考虑短剧的特殊要求。
3. **宏观优先 (Macro First)**：优先指出**剧情压缩**、**情节取舍**、**节奏把控**的大问题，忽略台词语病等琐碎问题。

# Inputs
${outlineContext ? `
1. 【分集大纲 (Blueprint/Outline)】：
${outlineContext}
` : ''}

${novelContext ? `
2. 【小说原文 (Novel Reference) - 改编参照基准】：
${novelContext}
` : ''}

3. 【待审核剧本 (Target)】：
${script}

# Audit Dimension Checklist (7大评估维度)

**评估核心原则：**
- **剧本自洽优先**：逻辑在剧本内部能自圆其说即可，不必与原著完全一致
- **正向改动加分**：如果改编比原著更精彩、更适合短剧，应该给高分
- **服务剧情优先**：人设改动只要能推动剧情、不违和，就是合理的

**⚠️ 短剧改编的核心要求：节奏压缩**
短剧的本质是"快节奏、高密度、强情绪"。小说改编短剧必须大幅压缩节奏：
- **合格的压缩**：5000字小说 → 1集短剧（约800字剧本）
- **优秀的压缩**：在压缩的同时保留甚至放大情绪爆点
- **失败的改编**：节奏和小说类似甚至更慢，这是致命伤！

## 1. 逻辑闭环 (Logic)
- 剧本内部的行为动机是否合理？因果链是否自洽？
- 即使与原著不同，只要剧本内逻辑能自圆其说，就应给高分
- 只有剧本内部出现逻辑漏洞、前后矛盾时才扣分

## 2. 情节推动 (Plot Efficiency)
- 每场戏是否有效推动剧情？信息密度是否足够？
- 关键转折点是否清晰有力？是否存在水时长？
- 原创情节如果能增强戏剧张力，应该加分
- **严重扣分项**：大段心理描写、冗长对话、无效场景

## 3. 钩子与节奏 (Hooks & Pacing) - ⚠️ 核心维度
- 开场前3秒是否抓人？结尾是否有悬念？
- **节奏压缩是否到位？** 这是最重要的评判标准！
- **致命问题（直接扣30分以上）**：
  * 节奏和小说原文类似或更慢
  * 大段照搬小说对话，没有精简
  * 铺垫过长，迟迟不进入冲突
  * 一集内容可以压缩到半集
- **优秀表现（可加分）**：
  * 原著3章内容精炼成1集，且保留核心冲突
  * 删除冗余铺垫，直接进入高潮
  * 用视觉动作替代大段心理描写

## 4. 人设一致性 (Character Consistency)
- 人物在剧本内是否前后一致？是否有突然的性格跳跃？
- 人设改动只要服务于剧情推动、不违和，就是合理的
- 即使与原著人设不同，只要剧本内人物立体、有魅力，就应给高分

## 5. 改编质量 (Adaptation Quality)
- 是否保留并放大了故事的核心魅力（爽点、虐点、金手指）？
- 正向改动（比原著更精彩）应该加分
- 视听化改编是否到位？文字转画面是否有创意？
- **严重扣分项**：照搬原著文字，没有视听化处理

## 6. 情绪带动 (Emotional Impact)
- 情绪点是否充分释放？爽感、虐感、甜感是否拉满？
- 情绪节奏是否有起伏？高潮点是否足够强烈？
- 如果情绪表达比原著更强烈，应该加分
- **严重扣分项**：情绪平淡、缺乏爆点、高潮被稀释

## 7. 画面表现 (Visual Presentation)
- 场景描写是否具有画面感？是否便于拍摄？
- 动作指示是否清晰、具体、可执行？
- 镜头语言是否丰富？视觉手法是否有效强化情绪？

# 评分标准（极度严格）

**节奏问题的严重性（必须严格执行）：**
- 如果剧本节奏和小说类似或更慢 → 总分不得超过55分
- 如果存在大段照搬小说对话 → "钩子与节奏"维度不得超过45分
- 如果一集内容明显可以压缩 → 至少扣20分

**总分标准：**
- **90-100分（现象级爆款）**：完美无缺，可遇不可求
- **85-89分（顶级商业水准）**：几乎无可挑剔，可直接投拍
- **80-84分（优秀水准）**：质量上乘，仅有极少量瑕疵
- **70-79分（良好水准）**：整体不错，但有明显可优化空间
- **60-69分（合格水准）**：基本可用，但问题较多需要修改
- **50-59分（勉强及格）**：问题明显，需要较大幅度修改
- **40-49分（不合格）**：问题严重，需要大幅重写
- **30-39分（严重不合格）**：基本失败，建议推翻重来
- **30分以下（完全失败）**：不具备改编价值

**维度评分要求：**
- 实事求是，好剧本各维度都可以高，差剧本各维度都可以低
- 敢于给极端分数：做得极好的给90+，做得极差的给40以下
- 每个维度独立评判，不要强行制造差异
- 大多数普通剧本应该在50-70分区间
- 如果"钩子与节奏"维度低于50分，总分不应超过60分
```

```typescript
// ============ 使用示例 ============

// 1. 配置 API
const apiConfig: APIConfig = {
  apiKey: 'your-api-key',

第1集

1-1 日/内/破旧出租屋

⊿ 简初夏蜷缩在角落，手里攥着一张泛黄的照片。

简初夏（独白）：三年了...我终于出来了。

⊿ 她站起身，眼神冰冷。

简初夏：陆景深，我回来了。

第一章 出狱

简初夏在监狱里待了整整三年。三年前，她被陆景深亲手送进了监狱...

      apiConfig,
      script,

    console.log('总评:', report.summary);

    // 输出致命问题

  } catch (error) {
    console.error('评价失败:', error);
  }
}
```

## 八、注意事项

### 8.1 全本通审注意事项

1. **模型选择**：推荐使用 GPT-4 Turbo / Claude 3 Opus 等长上下文模型
2. **超时处理**：长剧本（>5万字）需要设置较长超时时间（10-15分钟）
3. **上下文压缩**：如果原著/大纲过长，需要智能截取关键部分
4. **JSON 解析**：LLM 返回的 JSON 可能包含 markdown 标记，需要清理
5. **评分校准**：可以通过 few-shot 示例来校准评分标准
6. **批量处理**：多集剧本建议逐集评价，避免单次请求过大

### 8.2 分集精修注意事项

1. **上下文精准**：分集审稿需要精准的本集上下文（大纲、原文、人设）
2. **衔接检查**：第2集及以后必须提供上一集结尾用于衔接检查
3. **人设注入**：人物设定需要包含当前阶段的状态、心理、关系
4. **问题聚焦**：输出3-6个最关键问题，避免信息过载
5. **指令可执行**：修改建议必须是具体可执行的动作指令

# 分集精修 (Episode Audit) 完整框架

## 九、分集审稿函数

```typescript
// ============ 分集精修审稿函数 ============

export async function auditEpisodeScript(
  apiConfig: APIConfig,
  currentScript: string,           // 当前集剧本
  config: EpisodeAuditConfig       // 分集审稿配置
): Promise<AuditItem[]> {

  // 1. 构建审稿 Prompt
  const prompt = buildEpisodeAuditPrompt(currentScript, config);

  // 2. 根据剧本长度设置超时
  const timeout = calculateEpisodeTimeout(currentScript.length);

  // 3. 调用 API
  const response = await callLLMAPI(
    apiConfig,
    [{ role: 'user', content: prompt }],
    {
      temperature: 0.5,
      responseFormat: { type: 'json_object' },
      timeout: timeout
    }
  );

  // 4. 解析返回结果
  let items = JSON.parse(cleanJsonResponse(response.text));

  // 5. 处理可能的包装对象
  if (items && !Array.isArray(items) && items.items) {
    items = items.items;
  }
  if (!Array.isArray(items)) {
    items = [];
  }

  // 6. 格式化为标准问题列表
  return items.map((item: any, index: number) => ({
    ...item,
    id: Date.now().toString() + index,
    status: 'pending'
  }));
}

// 计算分集审稿超时时间
function calculateEpisodeTimeout(scriptLength: number): number {
  if (scriptLength > 150000) return 900000;   // 15分钟
  if (scriptLength > 100000) return 720000;   // 12分钟
  if (scriptLength > 50000) return 480000;    // 8分钟
  return 180000;                               // 3分钟
```

```typescript
export async function compareScripts(
  apiConfig: APIConfig,
  scripts: ScriptInput[]           // 待对比的剧本列表（至少2个）
): Promise<ComparisonReport> {

  // 1. 验证输入
  if (scripts.length < 2) {
    throw new Error("至少需要2个剧本进行对比");
  }

  // 2. 构建对比 Prompt
  const prompt = buildComparisonPrompt(scripts);

  // 3. 根据总长度设置超时
  const totalLength = scripts.reduce((sum, s) => sum + s.content.length, 0);
  const timeout = calculateComparisonTimeout(totalLength);

  // 4. 调用 API
  const response = await callLLMAPI(
    apiConfig,
    [{ role: 'user', content: prompt }],
    {
      temperature: 0.4,
      responseFormat: { type: 'json_object' },
      timeout: timeout
    }
  );

  // 5. 解析返回结果
  const result = JSON.parse(cleanJsonResponse(response.text));

  // 6. 添加时间戳并返回
  return {
    ...result,
    timestamp: Date.now()
  };
}

// 计算横向对比超时时间
function calculateComparisonTimeout(totalLength: number): number {
  if (totalLength > 150000) return 900000;   // 15分钟
```

## 二十、三种审稿模式完整对比

| 特性 | 全本通审 (Deep Audit) | 分集精修 (Episode Audit) | 横向对比 (Comparison) |
|------|----------------------|-------------------------|----------------------|
| **适用场景** | 整本剧本宏观评估 | 单集剧本精细打磨 | 多版本剧本择优 |
| **输入** | 全本剧本 + 原著 + 大纲 | 单集剧本 + 本集上下文 | 2个以上同集剧本 |
| **输出** | 7维度评分 + 批注列表 | 3-6个关键问题列表 | 排名 + 优缺点分析 |
| **评估维度** | 逻辑/情节/节奏/人设/改编/情绪/画面 | 主线/衔接/节奏/冲突/人设/视觉/台词 | 冲突前置/情绪价值/视觉化/台词效率 |
| **评分方式** | 0-100分 + 维度评分 | 严重程度 (high/medium/low) | 0-100分 + 排名 |
| **核心目标** | 发现问题、指导修改 | 精细打磨、逐条修复 | 选出最佳版本 |
| **推荐模型** | Claude Opus / GPT-4 Turbo | GPT-4 / Claude Sonnet | GPT-4 / Gemini Pro |
| **超时时间** | 3-30分钟（按长度） | 3-15分钟（按长度） | 3-15分钟（按总长度） |
